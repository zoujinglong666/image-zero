## 深度调试：图片分析 1305 全平台限流

**真实根因：** 智谱 AI `glm-4.6v-flash` 免费版触发全平台级别 429/1305 限流（"该模型当前访问量过大"），与代码无关，是服务端侧的全局并发配额满。

**修复架构（三层 fallback）：**
```
zai-sdk (zhipu) → 429重试3次 → SiliconFlow Qwen2-VL → 429重试3次 → OpenAI兼容层 → 降级
```

**新增：**
1. `analyzeWithSiliconFlow()` 方法，走 OpenAI 兼容 `https://api.siliconflow.cn/v1`，模型 `Qwen/Qwen2-VL-7B-Instruct`
2. `application.yml` 加 `siliconflow.api-key/base-url/analyze-model`
3. `backend/.env` 加 `SILICONFLOW_API_KEY=`（待填）

**待用户操作：** 注册 https://cloud.siliconflow.cn/account/ak，将 key 填入 `backend/.env` 的 `SILICONFLOW_API_KEY=`，然后重启后端即可。


### 根本原因
- `AiService.java` 之前只有 OpenAI 兼容层和 Gemini 两条路径
- `backend/.env` 中用 `OPENAI_API_KEY` 填了智谱 key、`OPENAI_BASE_URL` 指向 `https://open.bigmodel.cn/api/paas/v4`
- GLM-4.6V-Flash 通过 OpenAI 兼容层时，`content` 字段偶发为空/返回错误格式，触发 `buildFallbackAnalysis()` 降级

### 修复方案
1. **pom.xml** — 添加 `ai.z.openapi:zai-sdk:0.3.3`（智谱官方 Java SDK）
2. **AiService.java** — 新增 `analyzeWithZhipu()` 方法，使用 zai-sdk 原生调用 GLM-4.6V-Flash；`resolveProvider` 优先级改为 zhipu > openai > gemini
3. **application.yml** — 新增 `zhipu.api-key: ${ZHIPU_API_KEY:}` 配置项
4. **backend/.env** — 新增 `ZHIPU_API_KEY`（与已有 key 同值），图片分析走原生 SDK 更稳定
5. **backend/.env.example** — 同步更新示例文件
